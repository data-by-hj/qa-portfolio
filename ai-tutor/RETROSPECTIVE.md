# 회고

## 프로젝트를 시작하며
근로장학생으로 근무하며 가장 인상 깊었던 업무를 꼽으라면 단연코 교내 웹 서비스 테스트다.
단순한 근로 활동이 아닌 내가 다니는 학교 서비스의 품질에 기여하고 있다는 뿌듯함을 느낄 수 있었고,
QA라는 직무를 처음으로 경험해볼 수 있는 소중한 기회였다.

---

## 잘 된 점 (Keep)

- **버그 리포트 체계화**: 단순히 오류를 발견하는 데 그치지 않고, 테스트 절차 → 예상 결과 → 실제 결과 → 오류 내용 구조로 리포트를 작성하여 개발사에서 재현하기 쉽도록 정리하였다.
- **회귀 테스트 수행**: 1차에서 발견한 버그가 2차에서 실제로 수정되었는지 직접 재검증하며 단순 발견을 넘어 수정 확인하여 마무리하였다.
- **보안 관점 확장**: 기능 테스트에만 그치지 않고 XSS, API 토큰 노출, 세션 관리 등 보안 취약점까지 테스트 범위를 스스로 확장하였다.

---

## 아쉬운 점 (Problem)

- **사전 계획 없이 테스트 진행**: 처음 업무를 받았을 때 QA의 작업 절차를 충분히 이해하지 못한 채 무작정 테스트를 진행하였다. 실무에서는 테스트 계획 수립 → 테스트 케이스 작성 → 실행 순서로 체계적으로 진행한다는 것을 나중에야 인지하게 되었다.
- **테스트 케이스 분류 미흡**: 초반에는 기능별 카테고리 구분 없이 발견되는 대로 기록하여 누락되거나 중복된 항목이 생겼다. 처음부터 영역별로 분류했더라면 더 촘촘한 테스트가 가능했을 것이다.

---

## 배운 점 (Learn)

- **보안 개념 습득**: XSS(Cross-Site Scripting)가 무엇인지, API 토큰이 어떻게 처리되어야 하는지, 세션이 로그아웃 후 어떻게 무효화되어야 하는지를 직접 테스트하며 몸으로 익혔다.
- **QA 프로세스 이해**: 테스트는 발견이 아니라 계획에서 시작한다는 것을 배웠다. 테스트 계획 수립 → 테스트 케이스 작성 → 실행 → 버그 리포트 → 회귀 테스트로 이어지는 전체 흐름을 알게되었다.
- **LLM 서비스의 특수성**: 일반 웹 서비스와 달리 LLM 기반 서비스는 응답이 비결정적이라 동일한 입력에도 결과가 달라질 수 있다는 점, 그리고 컨텍스트 유지와 파일 참조 정확성이 핵심 품질 요소임을 알게 되었다.

---

## 다음에 적용할 것 (Try)

- **테스트 계획서 먼저 작성**: 다음 프로젝트에서는 테스트 시작 전 반드시 범위, 우선순위, 일정을 정의한 계획서를 먼저 작성한 후 진행할 것이다.
- **테스트 케이스 카테고리 분류**: 기능 / 보안 / UI·UX / 경계값으로 영역을 나눠 누락 없이 체계적으로 케이스를 설계할 것이다.
- **자동화 테스트 도입 시도**: 반복적으로 수행한 회귀 테스트를 Selenium 또는 Playwright로 자동화하여 효율을 높이고 싶다.
- **버그 관리 툴 활용**: 스프레드시트 대신 Jira나 Notion 데이터베이스를 활용해 버그 상태를 시각적으로 추적하는 방식을 적용해보고 싶다.
