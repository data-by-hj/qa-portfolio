# QA란 무엇인가?

## QA (Quality Assurance)의 정의

QA는 제품이 출시되기 전, 의도한 대로 동작하는지 검증하는 품질 보증이다.
단순히 버그를 찾는 것뿐만 아니라 사용자 관점에서 서비스 전반의 품질을 평가하고
문제를 체계적으로 기록하여 개발팀이 빠르게 수정할 수 있도록 돕는 역할을 한다.

---

## QA 엔지니어의 R&R

### Role

- QA 엔지니어는 개발자와 사용자 사이에서 서비스 품질을 **책임지는** 역할을 한다.
- 개발자가 만든 기능이 요구사항대로 동작하는지 확인하고, 사용자가 마주할 수 있는 문제를 사전에 발견하여 제품의 완성도를 높인다.
- 모두가 괜찮다고 하더라도 문제가 있다면 **적극적으로 논의**할 수 있어야 한다.
- 출시 후 발견되는 버그는 출시 전보다 훨씬 많은 비용이 발생한다. QA는 이 비용을 **사전에 최소화**하는 핵심 업무다.

### Responsibility

| 작업 | 설명 |
|------|------|
| 테스트 계획 수립 | 테스트 범위, 우선순위, 일정 정의 |
| 작업 명세서 검토 및 보증 | 기획 내용의 예외 사항 및 제안 검토 |
| 테스트 케이스 / 체크리스트 작성 | 기능별 시나리오 및 예상 결과 정의 |
| 테스팅 / 테스트 수행 | 기능 / 보안 / UI·UX / 경계값 등 다양한 관점 검증 |
| 이슈 등록 · 추적 · 관리 | 발견된 버그를 구조화하여 기록하고 수정 여부 추적 |
| 테스트 결과 보고 및 공유 | 테스트 결과를 관계 부서에 명확하게 전달 |
| 분석 및 제안서 작성 | 품질 개선을 위한 분석 및 제안 |
| QA 프로세스 개선 | 테스트 프로세스를 지속적으로 발전시키는 활동 |

> **테스트 vs 테스팅**
> - **테스트**: 특정 목적을 가지고 수행하는 개별적인 검증 활동 또는 절차
> - **테스팅**: 테스트를 포함하는 전체 과정 (계획 / 설계 / 실행 / 결과 분석 포함)

## QA 프로세스
마일스톤 시작 → 작업 명세서 검토 → 개발 → 빌드 → QA 수행 → 이슈 수정 및 검수 → 마일스톤 종료
| 단계 | 내용 |
|------|------|
| 마일스톤 시작 | 개발 기간에 따라 단계를 나누고 특정 목표점 설정, QA 일정 논의 |
| 작업 명세서 검토 | 기획 내용 확인, 예외 사항 및 제안 도출, 테스팅 시 필요한 정보 확인 |
| 개발 | 각 팀(서버 / 클라이언트 / 아트 / 사운드)에서 개발 진행 |
| 빌드 | 개발 완료 후 테스트 가능한 빌드 생성 |
| QA 수행 | 테스트 케이스 기반으로 기능 검증 및 이슈 등록 |
| 이슈 수정 및 검수 | 개발팀 수정 후 회귀 테스트로 수정 여부 재확인 |
| 마일스톤 종료 | 품질 기준 충족 시 해당 마일스톤 완료 |

> **마일스톤이란**
> 프로젝트를 여러 단계로 나눴을 때 각 단계의 목표 지점.
> 큰 프로젝트를 작은 단위로 쪼개어 관리하기 위해 사용한다.
